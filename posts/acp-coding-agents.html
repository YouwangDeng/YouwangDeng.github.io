ACP Is the Missing Protocol for AI Coding Agents

Every AI coding agent speaks its own language. Claude Code has -p --output-format stream-json. Cline has --act --json. Goose, Kiro, Gemini — each with their own CLI flags and output formats. The Agent Client Protocol makes them all speak the same one.

The N×M Problem

Right now, if you want to integrate M agents into N editors (or tools), you need N×M custom integrations. JetBrains needs a Cline plugin, a Claude plugin, a Goose plugin. VS Code needs the same. Zed needs the same. Every combination is a separate engineering effort.

We've seen this movie before. Before LSP, every editor had custom language support for every programming language. TypeScript worked in VS Code, sort of worked in Vim, barely worked in Emacs. LSP turned N×M into N+M — implement the protocol once per editor and once per language, and everything connects.

ACP is LSP for AI coding agents.

What It Actually Is

JSON-RPC 2.0 over stdio. That's it. An agent is a process that reads JSON from stdin and writes JSON to stdout. The protocol defines a handful of methods:

# 1. Client says hello
→ initialize {clientInfo, capabilities}
← {agentInfo, capabilities}

# 2. Create a workspace session
→ session/new {workspaceRoots, cwd}
← {sessionId}

# 3. Send a prompt
→ session/prompt {sessionId, prompt: [{type: "text", text: "..."}]}

# 4. Agent streams updates back
← session/update {agent_message_chunk}  # text
← session/update {tool_call}            # agent using a tool
← session/update {plan}                 # execution plan

# 5. Done
← session/prompt response {stopReason: "end_turn"}

The session is the key concept. Unlike one-shot CLI invocations, an ACP session persists. You send prompt #1, the agent works, you send prompt #2 — the agent has full conversation context. It remembers what files it created, what decisions it made, what you asked before.

Why This Matters Beyond Editors

The obvious use case is editors: JetBrains, Zed, Neovim get universal agent support. But ACP is more interesting than that.

Any program can be an ACP client. Not just IDEs. I implemented an ACP client in bash — 220 lines with named pipes and jq. It runs inside sage, my terminal-based agent orchestrator, and can talk to any ACP-compatible agent:

sage create worker --agent cline
sage create reviewer --agent goose
sage create planner --agent kiro

sage send worker "Build the REST API"
sage send worker "Now add authentication"  # same session, full context
sage send worker "Switch to PostgreSQL" --force  # cancel mid-task

One universal runtime for every agent. No custom parsers. No per-agent output format handling. Just the protocol.

What I Learned Implementing It

I added ACP support to sage as an experiment. Here's what surprised me:

1. Interactive TUI modes are a dead end

Running agents in interactive mode inside tmux and sending keystrokes via tmux send-keys doesn't work. Both Claude Code and Cline use full terminal UI frameworks with theme pickers, security prompts, and permission selectors that don't respond reliably to raw keystroke injection. ACP bypasses all of this — no TUI, just stdio JSON-RPC.

2. Persistent sessions change everything

With one-shot CLI modes (claude -p, cline --act), every task is a fresh process. The agent has no memory of previous work. If you want to iterate — "now add error handling to what you just built" — you need to re-explain the entire context.

With ACP sessions, follow-up prompts are just session/prompt into the same session. The agent remembers. This is the difference between sending emails to a contractor and having a conversation with a teammate.

3. The permission model differs per agent

Cline handles tool permissions internally — it rarely asks the client. Claude Code's ACP adapter sends session/request_permission with specific option IDs, and the response format needs a nested outcome object, not a simple boolean. Both also require cwd and mcpServers in session/new params, which aren't in the base spec. Plan for a few lines of agent-specific handling per agent, not zero.

4. You don't need Node.js for JSON-RPC

Bidirectional JSON-RPC over stdio sounds like it needs a proper runtime. It doesn't. Named pipes and file descriptors handle it cleanly in bash:

mkfifo "$fifo_in" "$fifo_out"
agent --acp < "$fifo_in" > "$fifo_out" &
exec 7>"$fifo_in"   # write to agent
exec 8<"$fifo_out"   # read from agent

echo '{"jsonrpc":"2.0",...}' >&7  # send
read -r response <&8              # receive

A complete ACP client — initialization, session management, event streaming, permission handling, cancel support — fits in ~220 lines of bash with zero dependencies beyond jq.

The Ecosystem Right Now

As of March 2026, 25+ agents implement ACP:

Verified working: Cline, Claude Code (via adapter), Goose, Kiro, Gemini CLI, Copilot CLI (preview)

Also listed: Augment Code, Cursor, OpenCode, Qwen Code, Mistral Vibe, OpenHands, Docker's cagent, and more via the ACP Agent Registry.

JetBrains and Zed co-develop the spec. The protocol is at version 1 — simple, stable, and doing exactly what it needs to.

ACP vs MCP vs A2A

Three protocols, three different problems:

  • MCP (Anthropic) — how agents connect to tools. Database access, web search, file systems. Agent-to-tool.
  • ACP (JetBrains/Zed) — how clients connect to agents. Editors, terminals, orchestrators talking to coding agents. Client-to-agent.
  • A2A (Google) — how agents talk to each other. Multi-agent collaboration and delegation. Agent-to-agent.

They're complementary, not competing. An ACP client sends a task to an agent, the agent uses MCP to access tools, and might delegate sub-tasks to other agents via A2A. Different layers of the same stack.

The Honest Downsides

ACP isn't all upside. Here's what gave me pause:

You lose direct control. With claude -p, I control exactly what flags get passed, what tools are allowed, what model is used. With ACP, the agent decides. The protocol has no standard way to say "use this model" or "don't use bash." You're trusting the agent's defaults, and each agent's defaults are different.

The spec is thin — implementations fill the gaps. ACP v1 is intentionally minimal. That means agents interpret it differently. Cline requires mcpServers in session/new. Claude's adapter needs a nested outcome object for permissions. These aren't bugs — they're the cost of a young spec where implementors made independent choices. You end up writing per-agent quirk handling anyway.

Session state is opaque. Once you start an ACP session, you can't inspect it. There's no "give me the conversation history" or "what files have you touched." The agent tracks this internally. If the agent process dies, your session context is gone. With one-shot CLIs, at least the state is explicit — it's whatever's on disk.

Debugging is harder. When cline --act --json breaks, I read the JSON output. When ACP breaks, I'm staring at bidirectional JSON-RPC over named pipes, trying to figure out which side dropped a message. The indirection adds a debugging layer that didn't exist before.

It's another protocol to maintain. We now have MCP for tools, ACP for agents, A2A for agent-to-agent. Each has SDKs, versions, breaking changes. If you're a small team building a coding tool, that's three specs to track. The "just parse CLI output" approach is ugly but has zero spec dependencies.

The Takeaway

ACP solves a real problem with a simple protocol. If you're building anything that talks to AI coding agents — an editor plugin, a CI pipeline, a terminal tool, an orchestrator — you should be speaking ACP instead of parsing custom CLI output formats.

The protocol is small enough to implement in a weekend. The ecosystem is big enough that your implementation immediately works with 25+ agents. And sessions give you something no amount of CLI flag parsing can: actual conversations with your agents.

Just go in with eyes open. The spec is young, implementations diverge, and you'll still need per-agent edge case handling. But the direction is right — and it's a lot better than maintaining N custom parsers.
