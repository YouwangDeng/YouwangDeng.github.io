I Built an Agent Framework in 1500 Lines of Bash

Every agent framework wants you to learn a new language, install a runtime, and buy into an ecosystem. I wanted to orchestrate AI coding agents. So I wrote 1500 lines of bash and called it sage.

The Problem

I had one agent building a frontend, another on the backend, and I was the bottleneck — switching terminals, copy-pasting results, losing track of what was where.

Every "agent framework" I found wanted Python + pip + virtualenv + a dozen packages, a custom DSL, a daemon, a database. I didn't need any of that. I needed to dispatch tasks to CLI agents and watch them work.

The Insight: Unix Already Solved This

Processes communicate through files. You manage them with signals. You observe them through terminals. What if an "agent framework" was just Unix primitives?

  • Agents = processes in tmux windows
  • Messages = JSON files in inbox directories
  • Tasks = status files + result files
  • Observability = tmux capture-pane

No daemon. No database. No Docker. No npm. Everything is a file you can cat.

What It Looks Like

sage create worker --runtime claude-code
sage send worker "Build a REST API with auth, tests, and docs"
sage peek worker   # watch it work in real-time

Three commands. Your agent runs in a tmux pane, writing code, calling tools. sage peek streams live output — tool calls as they happen, files as they're created.

Multiple agents? They orchestrate themselves:

sage create orch --runtime claude-code
sage send orch "Build a full-stack app. Delegate to sub-agents."

sage status
#  orch         running   45s
#    └─ frontend running   30s
#    └─ backend  running   28s

Agent going the wrong direction? Steer it:

sage steer orch "Use PostgreSQL instead of SQLite"
sage send worker "Actually, use React" --force  # cancel + redirect

The Architecture

~/.sage/
├── agents/worker/
│   ├── inbox/          # drop a JSON file = send a message
│   ├── workspace/      # agent writes files here
│   ├── results/        # task status + output
│   └── steer.md        # steering context
├── runtimes/           # bash, cline, claude-code, acp
├── trace.jsonl         # append-only event log
└── runner.sh           # the whole engine

The runner is a bash loop. Every 300ms it checks the inbox. Message found → source the runtime → call runtime_inject(). That's it.

Adding a new runtime is one file with two functions: runtime_start() and runtime_inject().

The Key Design Decision

Every agent framework I've seen makes the same mistake: they ask the LLM to manage state. "Remember to report your results." "Track your progress." LLMs forget. They hallucinate protocols. They skip steps.

In sage, everything that matters is mechanical. The runner writes task status, not the LLM. The engine tracks parent-child relationships. The runner captures results. Tracing is append-only JSONL written by runner code, not agent self-reporting.

The LLM does what it's good at — reasoning and writing code. The engine handles everything else.

Try It

# Install (pick one)
brew tap youwangd/sage && brew install sage
npm install -g @youwangd/sage
curl -fsSL https://raw.githubusercontent.com/youwangd/SageCLI/main/install.sh | bash

# Use it
sage init
sage create worker --runtime claude-code
sage send worker "Build something cool"
sage peek worker

Open source: github.com/youwangd/SageCLI. ~1500 lines of bash. MIT license. Read it in an afternoon.

Because the best agent framework is the one you can actually understand.
