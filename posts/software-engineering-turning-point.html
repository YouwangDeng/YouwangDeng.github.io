Software Engineering's Turning Point

Karpathy scored software developers 8-9 out of 10 on his AI exposure index. He's not wrong. But "exposed" doesn't mean "dead." It means the job is splitting in two — and the split is already happening.

The Split

One year ago, Karpathy coined "vibe coding" — casual, prompt-driven, good for throwaway projects. By February 2026 he declared it passé and introduced "agentic engineering": you don't write code, you direct agents that do.

That's not a prediction. It's a description of what's already real. A solo developer with Claude Code and a few agents can ship a complete SaaS product in a weekend — frontend, backend, database, deployment, docs. That workload used to be a small team for two or three weeks.

The job is splitting into two tracks:

Track 1: The Orchestrator. You decompose problems, dispatch agents, review output, steer direction. The skill is knowing what to build and whether it's right. You're the architect and the quality gate. The agents are the workforce.

Track 2: The Commodity. Writing CRUD endpoints, boilerplate configs, standard integrations. AI does this at 80% quality for 1% of the cost. Not perfect, but good enough fast enough that paying a human $150K for it stops making economic sense.

What the Numbers Say

Karpathy's analysis scored 342 US occupations by feeding BLS job descriptions into Gemini Flash. The scoring rubric has one killer heuristic: if the job can be done entirely from a home office on a computer, AI exposure is inherently high.

Software developers scored 8-9/10. Not because AI writes perfect code — it doesn't. But because the work product is purely digital, the feedback loop is tight (run it, see if it works), and the output is verifiable by machines (tests pass or they don't).

Compare that to an electrician (2-3/10). The work requires physical presence, manual dexterity, real-time judgment about physical systems. AI can't pull wire through conduit. It can, however, write your API handler in 30 seconds.

What AI Still Can't Do

High exposure doesn't mean full replacement. Here's what the agents still get wrong:

Taste. An agent will build exactly what you ask for. It won't tell you that what you're asking for is the wrong thing to build. Product sense — knowing which features matter, which abstractions will hold, which shortcuts will haunt you — that's still entirely human.

The weird bugs. When an ACP event loop won't cancel because read -t 1 keeps succeeding on a fast event stream — that requires understanding the actual system, not just prompting harder. Agents are great at generating code. They're terrible at debugging emergent behavior across system boundaries.

Judgment calls. Kill the process and restart fresh, or try to gracefully drain the event queue? That's an engineering decision that depends on context the agent doesn't have. The human who's been staring at the system for hours makes that call.

Cultural context. An AI translated "Oracle" as "神谕" (divine oracle) in a tech blog. Technically correct. Culturally tone-deaf. A native reader catches that instantly. The model doesn't even know it's wrong.

The Uncomfortable Part

Karpathy admits his ability to write code manually is atrophying. That's honest, and it's happening to all of us. The muscle memory fades when you stop exercising it.

He also predicts a "slopacolypse" — a flood of AI-generated code that's almost right but not quite. GitHub is already full of it. Someone still needs to know the difference between code that works and code that works correctly.

Here's the uncomfortable truth: the floor is rising. A single developer with agents can ship what used to take a team. That's great if you're the one developer. It's existential if you're one of the five who used to be on that team.

Junior roles are the most exposed. Not because juniors aren't valuable, but because "write this function given this spec" is exactly what agents do best. The entry ramp into the profession is getting steeper, and nobody has a good answer for that yet.

How to Be on the Right Side

The engineers who thrive will be the ones who move up the abstraction stack:

  • Learn to orchestrate, not just implement. Know how to break a project into agent-sized tasks, review what comes back, and steer when it drifts.
  • Go deeper on systems. Agents write application code well. They're weak on distributed systems, performance tuning, and debugging across boundaries. That gap is your moat.
  • Build taste. The hardest skill to automate is knowing what's worth building. Talk to users. Understand the business. Make product decisions.
  • Own the tools. The people building the orchestration layer — agent frameworks, protocols, developer infrastructure — are creating leverage, not consuming it.

Karpathy's autoresearch is the leading indicator. A 630-line Python script ran 700 autonomous experiments in two days and found improvements a 20-year veteran missed. The agents aren't coming for your job. They're coming for the parts of your job you thought only you could do.

The turning point isn't whether software engineering survives. It will. The turning point is whether you adapt to what it's becoming.
