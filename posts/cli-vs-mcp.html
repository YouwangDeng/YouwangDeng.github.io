CLI vs MCP: The Real Tradeoff Nobody's Talking About

Search GitHub for "MCP server" and you'll find thousands of repos. MCP server for PostgreSQL. MCP server for GitHub. MCP server for filesystem operations. MCP server for grep. Meanwhile, every AI coding agent already has a shell. So what's actually going on here?

The Explosion

In the last year, the MCP ecosystem went from a handful of reference implementations to over 10,000 servers on GitHub. There are MCP servers for everything: databases, APIs, file systems, Docker, Kubernetes, Slack, Gmail, weather, calculators, and yes — wrappers around CLI tools that agents could just run directly.

At the same time, every major AI lab shipped a CLI coding agent. Claude Code, Gemini CLI, Codex CLI, Cline, Goose, Aider, OpenCode, Kiro. All of them run in your terminal. All of them can execute shell commands. All of them can pipe, grep, curl, and jq their way through any task a Unix system can handle.

This creates a genuine question: if your agent has a shell, why does it need an MCP server?

What MCP Actually Does

MCP (Model Context Protocol) standardizes how AI agents discover and invoke tools. An MCP server exposes a set of capabilities — tools, resources, prompts — with typed schemas. The agent reads these schemas, knows what parameters each tool expects, and calls them via structured JSON-RPC.

# MCP: structured, typed, schema-driven
→ tools/call {name: "query_database", arguments: {sql: "SELECT ...", database: "prod"}}
← {content: [{type: "text", text: "42 rows returned..."}]}

# CLI: just run the command
$ psql -d prod -c "SELECT ..." --csv

Both get you the same result. The difference is in the contract.

The Case for CLI: Why Less Is More

The CLI-first argument is compelling, and it's rooted in 50 years of Unix philosophy:

Zero setup cost. Every tool your system already has — git, docker, curl, psql, kubectl, jq, grep, awk — is immediately available to any agent with shell access. No server to install, no schema to write, no process to manage. The agent just runs the command.

Token efficiency. When Claude Code needs to search your repo, it runs grep -rn "pattern" src/. That's 30 tokens. An MCP server for the same operation loads its entire tool schema into the system prompt — descriptions, parameter types, examples — permanently consuming context window space even when those tools aren't being used.

LLMs already know CLI. Models are trained on billions of lines of shell scripts, Stack Overflow answers, and man pages. Ask any LLM to write a find command and it'll nail it. Ask it to use your custom MCP server's search_files tool, and it needs to learn your specific schema first.

Composability is built in. Unix pipes are the original tool-chaining mechanism. grep pattern file | sort | uniq -c | sort -rn | head -10 is a pipeline that MCP can't express in a single call — you'd need multiple tool invocations with intermediate state management.

Self-correcting feedback loops. Exit codes and stderr give agents deterministic signals. Command failed? Read stderr, fix the command, retry. This loop is natural in shells and awkward through MCP's abstraction layer.

The Case for MCP: Why Structure Matters

So why does MCP exist at all? Because CLI has real limits that become painful at scale:

Structured output. psql returns tab-separated text. gh pr list returns formatted tables. Every CLI has its own output format that requires parsing. MCP returns typed JSON. When your agent needs to process the result programmatically — not just show it to a human — structured output eliminates an entire class of parsing bugs.

Discovery and documentation. An MCP server's tool list is a machine-readable API contract. The agent knows exactly what tools exist, what parameters they take, and what they return. With CLI, the agent either needs to already know the tool, or run --help and parse human-readable documentation — a lossy process.

Security boundaries. Giving an agent bash is giving it everything. MCP servers can scope access precisely: read-only database queries, specific GitHub repo operations, controlled file system access. In enterprise environments where AI agents need guardrails, MCP's permission model is the difference between "deploy it" and "absolutely not."

Remote and stateful services. Some tools don't have a CLI, or the CLI is inadequate. Your internal company API? A SaaS platform's webhook system? A database behind a VPN with connection pooling? MCP servers can wrap these cleanly. Shell commands can't curl their way through OAuth2 token refresh flows elegantly.

Multi-agent safety. When multiple agents share a system, raw shell access creates race conditions and conflicts. MCP servers can implement locking, queueing, and access control at the tool level — something that's nearly impossible to enforce through shell conventions alone.

The Real Problem: Context Pollution

Here's the insight most comparisons miss. The CLI vs MCP debate isn't really about capability — it's about context window economics.

Every MCP server you connect to an agent loads its tool schemas into the system prompt. Five MCP servers with 10 tools each? That's 50 tool definitions consuming thousands of tokens before the agent even looks at your code. This is "context pollution" — and it directly degrades the agent's reasoning quality.

CLI tools don't have this problem. They're invoked on demand. grep consumes zero context tokens until the agent actually decides to search for something. The tool's interface lives in the model's training data, not in the context window.

This is why Firecrawl, a company that ships both a CLI and an MCP server, explicitly argues that CLIs are better for agents. Their finding: teams that over-adopted MCP integrations are pulling back, opting for focused tool sets instead.

The pattern emerging in 2026 is clear: the best agent setups use few MCP servers, not many.

When to Use What

After building agent systems that use both approaches, here's the framework I've landed on:

Use CLI when:

  • The tool exists and is well-known (git, docker, curl, psql, kubectl)
  • The agent has shell access (most coding agents do)
  • Output is consumed by humans or simple text processing
  • You need composability (pipes, redirects, chaining)
  • You want zero setup overhead

Use MCP when:

  • The service has no CLI or a bad one (internal APIs, SaaS platforms)
  • You need structured, typed output for programmatic consumption
  • Security scoping is required (enterprise, multi-tenant, production databases)
  • The agent doesn't have shell access (web-based agents, sandboxed environments)
  • You need remote tool access across network boundaries

Don't wrap a CLI as an MCP server unless you're adding real value beyond what the CLI already provides — like access control, structured output, or remote invocation. A grep MCP server is strictly worse than grep.

The Convergence

The smartest tools are already converging on a hybrid approach:

  • Claude Code uses shell for local operations but supports MCP servers for extending into remote services
  • Gemini CLI runs local commands natively but connects to MCP servers for Google Search, databases, and external APIs
  • Codex CLI can act as an MCP server itself — other tools call into it, while it uses shell internally

The pattern: shell for local, MCP for remote. Shell for known tools, MCP for custom ones. Shell for quick operations, MCP for governed ones.

Nobody's going all-CLI or all-MCP. The winning architecture uses both, and the art is knowing which to reach for.

The Uncomfortable Truth

Most of those 10,000+ MCP servers on GitHub shouldn't exist. They're wrapping tools that agents already know how to use directly. They add latency, consume context, introduce dependencies, and create maintenance burden — all for capabilities the agent already has.

But the ones that should exist are genuinely transformative. An MCP server that gives an agent governed access to a production database? Essential. One that connects to your company's internal deployment API? Unlocks workflows that shell commands can't reach. One that provides real-time web search with structured results? Materially better than parsing curl output.

The mistake isn't MCP itself — it's the assumption that every tool needs to be an MCP server. Sometimes grep is just grep.
