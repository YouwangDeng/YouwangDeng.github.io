CLI vs MCP：没人聊的真正取舍

去 GitHub 搜「MCP server」，能翻出几千个仓库。PostgreSQL 的、GitHub 的、文件系统的，甚至还有给 grep 套壳的。但问题是——现在每个 AI 编程智能体都自带 shell。那这一切图什么？

井喷

短短一年，MCP 生态从几个参考实现暴涨到 GitHub 上一万多个服务器。数据库、API、文件系统、Docker、Kubernetes、Slack、Gmail、天气、计算器……应有尽有。其中不少，就是给 CLI 工具裹了层皮——而这些工具，智能体本来就能直接跑。

同一时间，各大 AI 实验室也在密集发布 CLI 编程智能体：Claude Code、Gemini CLI、Codex CLI、Cline、Goose、Aider、OpenCode、Kiro。清一色跑在终端里，能执行 shell 命令，pipe、grep、curl、jq 随便用，Unix 系统能干的活它们全能干。

于是一个真问题就来了：智能体都有 shell 了，还要 MCP 服务器干嘛？

MCP 到底在干什么

MCP（Model Context Protocol，模型上下文协议）要解决的是：让 AI 智能体用标准化的方式发现和调用工具。

一个 MCP 服务器对外暴露一组能力——工具、资源、提示词——每个都带类型化 schema。智能体读 schema，知道该传什么参数，然后走 JSON-RPC 调用。

# MCP：结构化、类型化、schema 驱动
→ tools/call {name: "query_database", arguments: {sql: "SELECT ...", database: "prod"}}
← {content: [{type: "text", text: "42 rows returned..."}]}

# CLI：直接跑命令
$ psql -d prod -c "SELECT ..." --csv

结果一样。差别在于「契约」的形式。

CLI 的底气：少即是多

CLI 优先派的论点很硬，背后是 50 年 Unix 哲学的底蕴：

零配置。系统上现成的工具——git、docker、curl、psql、kubectl、jq、grep、awk——只要智能体有 shell 权限，开箱即用。不用装服务器，不用写 schema，不用管进程。直接跑就完了。

省 token。Claude Code 要搜代码？跑一行 grep -rn "pattern" src/，30 个 token 搞定。换成 MCP 呢？光是加载工具 schema 到系统提示词——描述、参数类型、示例——就吃掉一大块上下文窗口，而且是常驻的，用不用都占着。

LLM 天生就会 CLI。模型在几十亿行 shell 脚本、Stack Overflow 和 man 手册上训练过。你让它写个 find 命令，信手拈来。但你让它用你自定义的 MCP 工具 search_files？它得先啃你那套 schema。

组合能力是祖传的。Unix pipe 是最早的工具链机制。grep pattern file | sort | uniq -c | sort -rn | head -10——一行搞定。MCP 要实现同样的效果？多次调用，外加中间状态管理。

天然的纠错循环。退出码和 stderr 给智能体提供的是确定性信号。命令挂了？读 stderr，改命令，重来。这在 shell 里浑然天成，隔了 MCP 的抽象层就变得别扭。

MCP 的底气：结构化凭什么重要

既然 CLI 这么香，MCP 为什么还有市场？因为 CLI 一旦上规模，短板就露出来了：

结构化输出。psql 吐的是制表符分隔文本，gh pr list 吐的是格式化表格。每个 CLI 输出格式都不一样，都得写解析逻辑。MCP 直接返回类型化 JSON。当智能体要拿结果做后续处理——不只是给人看——结构化输出直接干掉了一整类解析 bug。

可发现性。MCP 服务器的工具列表就是一份机器可读的 API 契约：有什么工具、要什么参数、返回什么。一目了然。CLI 呢？智能体要么提前知道这个工具存在，要么跑 --help 去啃人类写的文档——这个过程有损耗。

安全边界。给智能体一个 bash，等于把钥匙全交出去了。MCP 服务器能精确划定权限：只读数据库查询、限定 GitHub 仓库、受控的文件系统访问。在企业环境里，这是「可以上线」和「想都别想」之间的分水岭。

远程和有状态服务。有些东西压根没有 CLI，有的 CLI 又烂得不行。公司内部 API？SaaS 平台的 Webhook？VPN 后面带连接池的数据库？MCP 服务器能干净利落地封装。你想用 curl 优雅地搞 OAuth2 token 刷新？那画面太美。

多智能体协作安全。多个智能体共用一套系统，裸 shell 访问必然带来竞态和冲突。MCP 服务器可以在工具层面做锁、做队列、做访问控制——纯靠 shell 约定，几乎不可能。

真正的痛点：上下文污染

大多数对比文章漏掉了一个关键点。CLI vs MCP 之争，本质上不是能力之争——是上下文窗口的经济学。

每接一个 MCP 服务器，它的工具 schema 就会被注入系统提示词。接 5 个服务器，每个 10 个工具？那就是 50 个工具定义，智能体还没看你的代码，几千个 token 已经没了。这就是所谓的「上下文污染」——直接拉低推理质量。

CLI 没这个问题。按需调用，用到才花 token。grep 在智能体真正搜索之前，消耗的上下文 token 是零。工具怎么用，模型的训练数据里早就有了，不需要塞进上下文窗口。

Firecrawl 是一家同时做 CLI 和 MCP 服务器的公司，但他们自己公开说 CLI 对智能体更友好。他们观察到：MCP 堆太多的团队正在往回撤，改用精简工具集。

2026 年的趋势已经很明了：好的智能体配置，MCP 服务器宜少不宜多。

什么场景用什么

我自己搭过两种方案混用的智能体系统，踩完坑后总结出了一套判断框架：

该用 CLI：

  • 工具本身就是成熟的常见工具（git、docker、curl、psql、kubectl）
  • 智能体有 shell 权限（大多数编程智能体都有）
  • 输出是给人看的，或者做简单文本处理
  • 需要组合能力（pipe、重定向、链式调用）
  • 追求零配置，即插即用

该用 MCP：

  • 服务没有 CLI，或 CLI 太烂（内部 API、SaaS 平台）
  • 需要结构化、类型化的输出供程序消费
  • 有安全管控需求（企业、多租户、生产数据库）
  • 智能体没有 shell 权限（Web 端智能体、沙箱环境）
  • 需要跨网络边界远程调用工具

别把 CLI 包成 MCP 服务器——除非你确实在 CLI 之上加了真东西：访问控制、结构化输出、远程调用。给 grep 套个 MCP 壳？严格劣于直接用 grep。

殊途同归

聪明的工具已经在走混合路线了：

  • Claude Code：本地操作走 shell，远程服务走 MCP
  • Gemini CLI：本地命令原生执行，Google Search、数据库、外部 API 走 MCP
  • Codex CLI：自己就能当 MCP 服务器——别的工具调它，它内部用 shell

规律很简单：本地用 shell，远程用 MCP。熟悉的工具用 shell，定制的工具用 MCP。快操作用 shell，管控操作用 MCP。

没有人全押 CLI，也没人全押 MCP。赢的架构一定是两者兼用。关键在于——你得知道什么时候该伸哪只手。

不太好听的真话

GitHub 上那一万多个 MCP 服务器，大部分不该存在。它们包的工具，智能体本来就会用。多了一层延迟，多吃一块上下文，多一个依赖，多一份维护负担——换来的能力，智能体本来就有。

但真正该存在的那些，确实是变革级的。给智能体提供受控的生产数据库访问？刚需。接入公司内部部署 API？打通了 shell 够不到的工作流。提供结构化的实时搜索结果？体验碾压手动解析 curl 输出。

问题从来不在 MCP 本身——在于「什么都要包成 MCP 服务器」这种惯性思维。有时候，grep 就是 grep，别折腾了。
