我用 1500 行 Bash 写了一个智能体框架

每个智能体框架都想让你学一门新语言、装一套运行时、融入一个生态。而我只想编排几个 AI 编程智能体。于是我写了 1500 行 bash，取名 sage。

问题出在哪

当时的状态是：一个智能体写前端，另一个写后端，而我自己成了瓶颈——来回切终端、复制粘贴结果、完全记不住谁在干什么。

市面上的「智能体框架」无一例外：Python + pip + virtualenv + 一堆依赖，自定义 DSL，守护进程，数据库。我不需要这些。我只需要把任务分发给 CLI 智能体，然后看着它们干活。

关键洞察：Unix 早就解决了这个问题

进程通过文件通信，用信号管理，通过终端观察。如果「智能体框架」就是 Unix 原语呢？

  • 智能体 = tmux 窗口里的进程
  • 消息 = inbox 目录下的 JSON 文件
  • 任务 = 状态文件 + 结果文件
  • 可观测性 = tmux capture-pane

没有守护进程，没有数据库，没有 Docker，没有 npm。所有东西都是文件，cat 就能看。

用起来什么样

sage create worker --runtime claude-code
sage send worker "Build a REST API with auth, tests, and docs"
sage peek worker   # watch it work in real-time

三条命令搞定。智能体在 tmux 窗格里运行，写代码、调工具。sage peek 实时输出——工具调用、文件创建，全程可见。

多个智能体？它们会自行协调：

sage create orch --runtime claude-code
sage send orch "Build a full-stack app. Delegate to sub-agents."

sage status
#  orch         running   45s
#    └─ frontend running   30s
#    └─ backend  running   28s

智能体跑偏了？随时纠正：

sage steer orch "Use PostgreSQL instead of SQLite"
sage send worker "Actually, use React" --force  # cancel + redirect

架构

~/.sage/
├── agents/worker/
│   ├── inbox/          # drop a JSON file = send a message
│   ├── workspace/      # agent writes files here
│   ├── results/        # task status + output
│   └── steer.md        # steering context
├── runtimes/           # bash, cline, claude-code, acp
├── trace.jsonl         # append-only event log
└── runner.sh           # the whole engine

runner 就是一个 bash 循环。每 300ms 检查一次 inbox。发现消息 → 加载 runtime → 调用 runtime_inject()。没了。

新增一个 runtime 只需要一个文件，里面写两个函数：runtime_start() 和 runtime_inject()。

最关键的设计决策

我见过的每个智能体框架都犯同一个错误：让 LLM 管理状态。「记得汇报你的结果。」「记录你的进度。」但 LLM 会遗忘，会幻觉出各种协议，会跳过步骤。

在 sage 里，一切重要的事都是机械化的。任务状态由 runner 来写，不是 LLM。父子关系由引擎来追踪。结果由 runner 捕获。Trace 日志是 runner 代码写入的 append-only JSONL，而不是智能体的自我汇报。

LLM 只做它擅长的事——推理和写代码。其余的交给引擎。
