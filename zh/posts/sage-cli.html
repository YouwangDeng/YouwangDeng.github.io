1500 行 Bash，一个智能体框架

市面上的智能体框架，个个都想让你学新语言、装运行时、绑定生态。我只想编排几个 AI 编程智能体而已。于是写了 1500 行 bash，叫它 sage。

痛点

我日常用 Claude Code、Cline 这些 AI 编程智能体干活，但总觉得别扭。一个写前端，一个写后端，我自己反倒成了瓶颈——来回切终端，复制粘贴，记不清谁在干什么。

我去看了看现有方案。每个所谓的「智能体框架」都要我：

  • 装 Python + pip + virtualenv + 一堆包
  • 学一套自定义 DSL 来描述工作流
  • 跑守护进程或服务
  • 把状态塞进数据库
  • 接受一套两周后必然想掀桌的架构设计

这些我都不需要。我的需求就一句话：把任务丢给 CLI 智能体，看它们干活。完事。

灵感

Unix 五十年前就把进程编排玩明白了。进程靠文件通信，靠信号管理，靠终端观察。

那「智能体框架」是不是也可以就是……Unix 原语？

  • 智能体 = 跑在 tmux 窗口里的进程
  • 消息 = 往 inbox 目录丢 JSON 文件
  • 任务 = ID + 状态文件 + 结果文件
  • 编排 = 智能体创建智能体
  • 可观测性 = tmux capture-pane

不用守护进程。不用数据库。不用 Docker。不用 npm。所有东西都是文件，cat 就能看。

sage 长什么样

建个智能体，丢个任务，看它跑：

sage create worker --runtime claude-code
sage send worker "Build a REST API with auth, tests, and docs"
sage peek worker   # watch it work in real-time

三条命令。智能体在 tmux 面板里写代码、调工具。sage peek 能实时看到它在干嘛——工具调用一条条流出来：

⚡ peek: worker

Runner log:

  I'll create a professional REST API for you...
    → ToolSearch
    → TodoWrite
    → Write
    → TodoWrite
    → Write

要跑多个智能体？编排起来：

sage create orch --runtime claude-code
sage send orch "Build a full-stack app. Delegate to sub-agents."

sage status
#  orch         claude-code  running   45s
#    └─ frontend claude-code  running   30s
#    └─ backend  claude-code  running   28s

智能体跑偏了？随时拉回来，不用重头开始：

# Soft steer — guidance for next task
sage steer orch "Use PostgreSQL instead of SQLite"

# Hard steer — nuclear option: stop everything, restart with context
sage steer orch "Switch to Go" --restart

--restart 是级联的——停掉所有子智能体、停掉编排器、写入引导上下文、把进行中的任务重新排队、重启。编排器按需重建子智能体。一条命令。

架构（简单到离谱）

~/.sage/
├── agents/worker/
│   ├── inbox/          # drop a JSON file = send a message
│   ├── workspace/      # agent writes files here
│   ├── results/        # task status + output (mechanical)
│   └── steer.md        # steering context (injected into prompts)
├── runtimes/           # bash, cline, claude-code
├── trace.jsonl         # append-only event log
└── runner.sh           # 300ms polling loop, that's the whole engine

Runner 就是个 bash 循环。每 300ms 扫一次 inbox，有消息就加载对应 runtime（claude-code、cline 或 bash），调 runtime_inject()。没了。这就是全部架构。

想加个新 runtime——比如 Gemini CLI 或 Aider？写一个文件，实现两个函数：runtime_start() 和 runtime_inject()。全部代码，一个下午读完。

机械化 > 行为化

这是我反复验证的核心设计原则。市面上的智能体框架都犯同一个毛病：让 LLM 管状态。「记得汇报结果。」「追踪你的进度。」「做完了通知编排器。」

LLM 会忘事。会编造协议。会跳步骤。在一个概率模型的「行为承诺」上建可靠系统？不可能的。

sage 里，所有关键操作都是机械化的：

  • 任务状态？Runner 写 queued → running → done。不是 LLM 写的。
  • 父子关系？引擎通过 SAGE_AGENT_NAME 环境变量记录。不靠智能体自觉。
  • 结果输出？Runner 捕获并写文件。不靠提示词里的一句嘱咐。
  • 链路追踪？Runner 代码写的 append-only JSONL。不靠智能体自我汇报。

LLM 只干它擅长的事：推理，写代码。剩下的，引擎全包。

实时流式输出（最难啃的骨头）

工程上最折腾的问题：怎么拿到实时输出。claude -p（print 模式）执行时会吞掉所有终端输出。stdout 是 TTY 也好，管道也好，套 script(1) 也好，都没用。Print 模式攒着所有内容，最后一次性吐出来。

我把能试的都试了：tee、tail -f、script --flush、进程替换、tmux pipe-pane。全是死胡同——CLI 在工具调用期间压根不往 stdout 写东西。

最后的解法：--output-format stream-json --verbose。Claude 会实时吐 JSONL 事件——每个工具调用一条，每段文本一条。用 while read 循环逐条解析，打印到终端：

cat "$prompt" | claude -p --output-format stream-json --verbose \
  --allowedTools "Bash(*)" "Write(*)" "Read(*)" "Edit(*)" \
  | while IFS= read -r line; do
    # parse JSON event, print tool calls + text
  done

Cline 用 --json 也是同理。每个 CLI 的事件格式不同，但套路一样：结构化事件过解析器，输出到终端。

收获

约束即特性。「只用 bash、jq 和 tmux」听着很受限，实则是解放。没有依赖地狱。没有版本冲突。没有构建步骤。整个工具就一个文件——读得懂、看得透、改得动。

文件就是万能 API。所有工具都会读写文件。目录里的 JSON 就是最朴素的消息总线。重启不丢，cat 能查，ls -la 能调试，零基础设施。

可观测性不是锦上添花。第一版没有 peek，没有 trace，没有流式输出。跟蒙眼开车一样。提交任务，然后等。30 秒搞定？还是卡了 5 分钟？完全不知道。加上实时输出后体验天翻地覆——从「提交后祈祷」变成了「提交后围观」。

LLM 是系统里最靠不住的环节。不是说它差——它写代码的能力惊人。但遵守协议这事，它真的不行。每次让智能体「记住」要做什么（汇报、清理、通信），它都会忘。把这些从行为约定改成机械执行，是可靠性提升最大的一刀。
