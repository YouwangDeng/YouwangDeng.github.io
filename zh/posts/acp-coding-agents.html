ACP：AI 编程智能体一直缺的那个协议

现在每个 AI 编程智能体都在自说自话。Claude Code 用 -p --output-format stream-json，Cline 用 --act --json，Goose、Kiro、Gemini 各有一套 CLI 参数和输出格式。Agent Client Protocol 做的事情很简单：让它们说同一种语言。

N×M 困局

假设你有 M 个智能体、N 个编辑器，想让它们互通，你需要写 N×M 个适配。JetBrains 得做 Cline 插件、Claude 插件、Goose 插件。VS Code 再来一遍。Zed 再来一遍。每个排列组合都是一次独立的工程量。

这剧本似曾相识。LSP 出现之前，每个编辑器都得自己做语言支持。TypeScript 在 VS Code 里体验不错，到了 Vim 勉强能用，Emacs 里基本抓瞎。LSP 把 N×M 降成了 N+M——编辑器实现一次协议，语言服务实现一次，就全通了。

ACP 就是 AI 编程智能体领域的 LSP。

它到底是个啥

说白了，就是跑在 stdio 上的 JSON-RPC 2.0。智能体是一个进程：从 stdin 读 JSON，往 stdout 写 JSON。协议就定义了这么几个方法：

# 1. Client says hello
→ initialize {clientInfo, capabilities}
← {agentInfo, capabilities}

# 2. Create a workspace session
→ session/new {workspaceRoots, cwd}
← {sessionId}

# 3. Send a prompt
→ session/prompt {sessionId, prompt: [{type: "text", text: "..."}]}

# 4. Agent streams updates back
← session/update {agent_message_chunk}  # text
← session/update {tool_call}            # agent using a tool
← session/update {plan}                 # execution plan

# 5. Done
← session/prompt response {stopReason: "end_turn"}

核心概念是会话（session）。跟一次性的 CLI 调用不同，ACP 的会话是持久的。你发第一条 prompt，智能体去干活；你发第二条，智能体还记得之前的一切——创建了哪些文件、做了什么决策、你之前说了什么。

价值不止于编辑器

最直观的场景当然是编辑器：JetBrains、Zed、Neovim 都能获得通用的智能体支持。但 ACP 的想象空间远不止这些。

任何程序都能当 ACP 客户端。不限于 IDE。我用 bash 写了一个 ACP 客户端——总共 220 行，靠 named pipe 加 jq 就跑起来了。它跑在 sage（我写的终端智能体编排工具）里面，能跟任何支持 ACP 的智能体对话：

sage create worker --agent cline
sage create reviewer --agent goose
sage create planner --agent kiro

sage send worker "Build the REST API"
sage send worker "Now add authentication"  # same session, full context
sage send worker "Switch to PostgreSQL" --force  # cancel mid-task

一套运行时通吃所有智能体。不用写定制解析器，不用处理各家输出格式的差异。协议就是接口。

实现过程中踩的坑

我在 sage 里加了 ACP 支持，算是一次实验。有几个发现出乎意料：

1. 交互式 TUI 是死路一条

在 tmux 里跑智能体的交互模式，然后用 tmux send-keys 模拟按键——别想了，走不通。Claude Code 和 Cline 都用了完整的终端 UI 框架，里面有主题选择、安全提示、权限弹窗，靠注入按键根本控制不住。ACP 直接绕过了这层——没有 TUI，只有 stdio JSON-RPC。干净利落。

2. 持久会话是质变

一次性 CLI 模式下（claude -p、cline --act），每次任务都起一个新进程。智能体对之前干过的事毫无记忆。想迭代？比如「给刚才写的代码加上错误处理」？那得把整个上下文从头讲一遍。

ACP 会话不同。后续指令直接往同一个会话发 session/prompt，智能体全都记得。打个比方：一次性 CLI 像给外包发邮件，ACP 会话像跟队友当面聊。

3. 各家权限模型各有一套

Cline 的工具权限在内部处理，很少问客户端。Claude Code 的 ACP 适配器会发 session/request_permission，带着特定的选项 ID，返回格式得是嵌套的 outcome 对象，不是简单的布尔值。两家还都要求在 session/new 里传 cwd 和 mcpServers，这些在基础规范里根本没提。所以别指望零适配——每个智能体多少都要写几行特殊处理。

4. 跑 JSON-RPC 不需要 Node.js

基于 stdio 的双向 JSON-RPC，听起来得上个正经运行时？其实不用。bash 里用 named pipe 加文件描述符就能搞定：

mkfifo "$fifo_in" "$fifo_out"
agent --acp < "$fifo_in" > "$fifo_out" &
exec 7>"$fifo_in"   # write to agent
exec 8<"$fifo_out"   # read from agent

echo '{"jsonrpc":"2.0",...}' >&7  # send
read -r response <&8              # receive

一个完整的 ACP 客户端——初始化、会话管理、事件流、权限处理、取消支持——大约 220 行 bash，除 jq 外零依赖。

目前的生态

截至 2026 年 3 月，已有 25+ 个智能体实现了 ACP：

实测可用的：Cline、Claude Code（通过适配器）、Goose、Kiro、Gemini CLI、Copilot CLI（预览版）

也在列表上的：Augment Code、Cursor、OpenCode、Qwen Code、Mistral Vibe、OpenHands、Docker 的 cagent 等等，完整名单见 ACP Agent Registry。

JetBrains 和 Zed 在联合推进规范。协议目前是 v1——简洁、稳定，做好了它该做的事。

ACP vs MCP vs A2A

三个协议，各管一摊：

  • MCP（Anthropic）——智能体怎么接工具。数据库、搜索、文件系统。智能体→工具。
  • ACP（JetBrains/Zed）——客户端怎么接智能体。编辑器、终端、编排器跟编程智能体对话。客户端→智能体。
  • A2A（Google）——智能体之间怎么协作。多智能体配合与任务委派。智能体→智能体。

三者互补，不是竞品。ACP 客户端把任务交给智能体，智能体通过 MCP 调用工具，还可能通过 A2A 把子任务分派给别的智能体。是同一个技术栈的不同层。

说说缺点

ACP 不是没有问题。让我犹豫的地方有这些：

你交出了控制权。用 claude -p 的时候，传什么参数、开哪些工具、用哪个模型，全在我手里。换成 ACP，这些都由智能体自己决定。协议里没有标准的方式说「用这个模型」或「别碰 bash」。你只能信任智能体的默认配置——而每家的默认配置都不一样。

规范太薄，全靠实现补。ACP v1 是故意做得极简的。代价就是各家理解不同。Cline 要求 session/new 里带 mcpServers。Claude 的适配器在权限响应里需要嵌套的 outcome 对象。这些不是 bug，是年轻规范下各实现者自行其是的结果。到头来你还是要写一堆特定智能体的兼容代码。

会话状态是个黑箱。ACP 会话一旦开始，你就看不到里面发生了什么。没有接口查对话历史，也没法问「你改了哪些文件」。所有状态都在智能体进程内部。进程一挂，上下文就没了。用一次性 CLI 至少状态是透明的——磁盘上有什么就是什么。

调试变难了。cline --act --json 出了问题，我直接看 JSON 输出就行。ACP 出了问题，我得盯着 named pipe 上来回跑的 JSON-RPC 消息，猜到底是哪头掉了包。多了一层间接抽象，就多了一层调试成本。

又多了一个要跟的协议。现在是 MCP 管工具、ACP 管智能体、A2A 管智能体间通信。每个都有 SDK、有版本、有 breaking change。小团队做编程工具，光协议就得跟三份。「直接解析 CLI 输出」的方式虽然丑，但好歹零规范依赖。

总结

ACP 用一个简单的协议解决了一个实打实的问题。如果你在做任何需要跟 AI 编程智能体打交道的东西——编辑器插件、CI 流水线、终端工具、编排系统——你该用 ACP，而不是去解析各家五花八门的 CLI 输出。

这个协议小到一个周末就能实现完。生态又足够大，你的实现一落地就能对接 25+ 个智能体。而会话机制给了你一样靠解析 CLI 永远得不到的东西：跟智能体进行真正的多轮对话。

当然，别盲目乐观。规范还年轻，各家实现有分歧，边界情况的特殊处理少不了。但方向没错——而且总比维护 N 个自定义解析器强得多。
